# AI Panel Studio — 产品需求文档 (PRD)

> 版本: v1.0 | 日期: 2026-06-26 | 状态: Draft

---

## 1. 背景与业务需求

### 1.1 产品定位

「AI Panel Studio」是一款"AI 圆桌讨论"本地 Web 应用：

- 用户输入任意讨论话题并指定参会专家人数（默认 4 人）
- 系统调用大模型动态生成一组主持人与专家嘉宾
- 展示每位嘉宾的姓名、职业、Title、立场与专属颜色标识
- 用户确认阵容后进入演播厅讨论模式
- 观看一场由 AI 驱动、实时推进的圆桌讨论

**核心价值主张**：让任何人都能瞬间召集一支"虚拟智库"，围绕任意议题展开深度碰撞——主持人负责开场、追问、串联与收尾，专家之间自主"举手/抢答/补充/反驳"，讨论过程中持续生成共识与分歧，最终呈现一场沉浸式的 AI 圆桌实况。

### 1.2 目标用户画像

| 用户群体 | 特征描述 | 核心诉求 |
|---|---|---|
| 知识工作者与研究人员 | 需要快速对一个议题进行多角度审视，时间有限 | 在几分钟内获得来自不同视角的深度分析与争辩摘要 |
| 产品/决策人员 | 在方案评审或战略讨论前希望预演各方立场与潜在争议 | 用 AI 圆桌发现盲点、检验假设，提升决策质量 |
| 教育与内容创作者 | 需要将复杂议题结构化呈现给受众，追求沉浸感与叙事性 | 获得一场可录制、可回放的"虚拟专家访谈"作为内容素材 |

### 1.3 用户痛点分析

1. **视角单一**：使用单一 AI 对话获得的分析往往立场单一，缺乏真实辩论中"对抗性思考"所带来的深度
2. **异步割裂**：多次单独询问不同角色的 AI，上下文割裂，观点之间无法真正产生互动与碰撞
3. **过程不可见**：AI 输出通常以最终结论呈现，缺乏"思考过程"和"观点演化"的实时感与沉浸感
4. **议题管理混乱**：用户希望同时探索多个议题，但不同讨论的上下文、进展与结论难以并行隔离管理

### 1.4 核心功能要求

1. **首页**：展示所有正在进行的讨论列表；用户可加入观察已有讨论，也可发起新讨论
2. **嘉宾生成**：用户输入话题与专家人数后，系统调用大模型动态生成主持人+专家阵容，展示姓名、职业/Title、立场、专属颜色标识，由用户确认后进入演播厅
3. **演播厅模式**：
   - 主持人负责开场、追问、串联和总结
   - 专家根据当前 transcript 自主决定发言顺序（举手/抢答/补充/反驳），每次发言控制在 1-2 句
   - 不允许机械式轮流发言
4. **专家状态小窗**：每位专家有独立小窗，实时显示 agent 运行状态（待机/准备发言/发言中），当前关注点或公开思考摘要；不展示真实的隐藏 chain-of-thought
5. **实时共识与分歧**：讨论过程中持续从发言中提炼共识与分歧，实时更新对应区域，不等到讨论结束后才生成
6. **现场 Transcript**：显示发言人姓名与职业/Title，用与专家一致的色块区分；不显示"举手"等内部事件
7. **总结**：结束时主持人以自然语言进行总结，禁止将 JSON 原文显示到页面上

## 2. 技术约束

| 约束项 | 要求 |
|---|---|
| 运行环境 | 本地运行，前后端分离 |
| AI API Key | 仅存放后端环境变量，不得暴露到浏览器端 |
| 多讨论并行 | 不同讨论的状态、事件流、transcript、共识与分歧必须互相隔离 |
| 实时更新 | SSE、WebSocket 或等价机制 |
| UI | 中文界面，响应式布局；各区域容器内独立滚动，不依赖整页滚动 |
| 数据库 | SQLite |

---

## 3. 开发约束

- 禁止"一键生成"：严禁使用单一 Prompt 直接生成整个项目代码
- 开发范式：SDD → DDD → TDD → E2E
- 原创性：必须从 0 到 1 创建
- 工具：主体开发使用 Claude Code + DeepSeek V4 Pro 模型
